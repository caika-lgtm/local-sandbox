## Current known blockers

- Windows VM startup now attempts direct Linux boot with the M06 virtio-serial control chardev enabled. It runs QEMU discovery/preflight, builds WHPX direct-boot argv, launches through the existing QEMU supervisor, connects the QEMU-created control pipe during boot, captures serial/stdout/stderr/preflight/status artifacts, and returns success only after receiving a valid LocalSandbox `GuestReady` protocol frame over the established control stream. Non-interactive exec is implemented for Windows through the platform-neutral `Sandbox::exec` / SDK `exec` path. M09 added guest file transfer primitives over the same serialized virtio-serial control path plus Windows-only `lsb-vm::Sandbox::copy_from_host` / `copy_to_host` helpers. M10 uses those helpers for staged copy/import overlay mounts: host directory sources are snapshotted into guest staging, guest writes stay isolated in the guest overlay, and explicit export uses the existing copy-out path. M11 adds host-to-guest port forwarding over a separate private virtio-serial guest channel with host listeners bound to `127.0.0.1`. M12 adds policy-mediated allow-net through a LocalSandbox-owned proxy stream attachment. M13 adds Windows checkpoint/store MVP semantics through immutable base images, per-instance qcow2 writable overlays, and flattened qcow2 checkpoint artifacts. M14 adds Windows x86_64 Node packaging metadata and SDK-backed native binding wiring. Unrestricted direct guest networking and CAS/NBD checkpoint transport remain unsupported.
- QEMU 11.0.50 on the self-hosted Windows WHPX runner blocks boot until a host client connects to the `-chardev pipe` endpoint. M06 now connects the pipe immediately after QEMU process start and stores the established stream on the running boot object; later `PlatformVm::connect_control()` clones that stream. See D021.
- Streaming `spawn`, interactive shell, watch, concurrent forwarding sessions, unrestricted direct networking, and CAS/NBD checkpoint transport remain later milestones. Windows streaming exec stdin/kill returns a precise unsupported error until a muxed transport exists; non-interactive exec closes guest stdin explicitly through the backward-compatible `ExecRequest.stdin_closed` field. M11 port forwarding intentionally serializes active Windows forwarding sessions over the dedicated forwarding channel until a real mux/session model exists. The Windows SDK `checkpoint()` path currently stops the VM before flattening the active qcow2 disk into a checkpoint artifact; this preserves checkpoint contents for the MVP but is not live checkpointing. `lsb-guest` emits `GuestReady` only on the Windows virtio-serial transport path; existing macOS/vsock behavior remains unchanged.
- Full `cargo check --workspace --target x86_64-pc-windows-msvc` from this macOS host is blocked by external Windows C/assembler tooling for transitive crates (`blake3` needs `ml64.exe`; `ring` needs Windows/MSVC headers such as `assert.h`; `aws-lc-sys` needs Windows headers such as `windows.h`; `boring-sys` needs a Visual Studio generator). The self-hosted Windows/MSVC check lane passed for M13 on run `28739318686` at commit `eaadb69`.
- The current safe host probe verifies target OS/arch and can report a supplied Windows major version. The standard host implementation does not yet query the native Windows build number without adding Windows API or registry probing.

The root package NAPI targets include `x86_64-pc-windows-msvc`; npm root `os`/`cpu` metadata cannot express OS/CPU pair exclusions precisely, so unsupported Windows ARM64/IA32 remain absent from platform packages and must fail through package/native-loader errors rather than published native artifacts.
- 2026-07-05: The Node native crate links `lsb-sdk` on Windows x86_64. Missing QEMU/WHPX/runtime-asset failures after native load should be Rust backend/preflight errors from `Sandbox.start()`, not generic native-module load errors. The generated NAPI loader is patched after build so missing Windows x64 native package errors name `@local-sandbox/lsb-nodejs-win32-x64-msvc` / `lsb-nodejs.win32-x64-msvc.node`, while unsupported Windows non-x64 hosts report that Windows Node support is limited to `win32-x64-msvc`.
- 2026-07-05: Node binding errors now join `anyhow` error chains before constructing `napi::Error`, so Windows backend preflight messages such as invalid `LSB_QEMU`, missing WHPX/QEMU tooling, qemu-img failures, and captured diagnostics paths remain visible to JavaScript callers.
- 2026-07-05: M13 default Windows storage, unless `LSB_STORAGE=direct` is set, creates a private per-instance `root.qcow2` overlay with `qemu-img create -f qcow2 -F <raw|qcow2> -b <source>`. The source is either the immutable base rootfs, a flattened Windows qcow2 checkpoint, or a legacy direct `.ext4` checkpoint. The Windows QEMU backend detects `.qcow2` roots and passes `format=qcow2`; `.ext4` debug disks remain `format=raw`.
- 2026-07-05: M13 checkpoint save uses `qemu-img convert -O qcow2 <active-overlay> <checkpoint>.qcow2` and writes versioned JSON metadata only after conversion succeeds. Checkpoints are listed only when both metadata and disk are present, delete removes `.json`, `.qcow2`, and legacy partial `.ext4`/`.idx` files, restore never mutates the checkpoint source, and CAS `.idx` restore on Windows returns a typed unsupported-mode error. This is a Windows MVP and not CAS/NBD parity.
- 2026-07-04: M10 mount support is snapshot/import only. The Windows backend does not attach QEMU shared-directory devices for mounts, does not use VirtioFS/9p/vfat, does not implement live file watching or bidirectional synchronization, and rejects direct writable host mounts. Host source validation reuses the M09 Windows path policy, including canonicalization where possible, traversal protection, missing-source/type checks, conservative reparse/symlink/hardlink handling, and case-insensitive collision checks. Mount imports are re-planned immediately before copying into the guest so source-tree changes after `Sandbox::build()` are revalidated at import time. Each planned file carries Windows file identity metadata and is reopened through a checked handle path before bytes are read, so file replacement, parent symlink/junction replacement to a different file, hardlink introduction, length changes, and final-component reparse points fail closed before import.
- 2026-07-05: M10 Windows path validation intentionally accepts `\\?\C:\...` drive-verbatim paths because `std::fs::canonicalize` produces that shape on Windows for CLI and Node mount sources. UNC verbatim paths and device paths remain unsupported. Windows mount targets under, at, or above `/tmp/lsb/mounts` are reserved so the imported lowerdir staging tree cannot be self-shadowed by a user mount target.
- 2026-07-05: M12 Windows networking is deny-by-default. Without allow-net, QEMU argv emits `-nic none` and no `-netdev`. With existing allow-net/proxy configuration, the host starts a LocalSandbox-owned loopback listener and QEMU connects with `-netdev stream,id=lsbproxy0,server=off,addr.type=inet,addr.host=127.0.0.1,addr.port=<proxy-port>` plus `virtio-net-pci`; diagnostics redact the ephemeral port and generated local proxy MAC. Legacy fd/socketpair attachments, non-loopback proxy endpoints, QEMU user networking, `hostfwd`, TAP, bridge, and NAT remain unsupported for the Windows product path.
- 2026-07-05: M12 preserves host-side secret semantics. Guest environment values are placeholders, not literal host secrets; `SecretConfig` debug output redacts literal values; configured allowlists are enforced before proxy upstream connect and before secret substitution. Explicit host allowlists block direct-IP, missing-domain, non-allowlisted-domain, and forged allowed Host/SNI-to-arbitrary-IP traffic by requiring the policy-visible domain to match a recent proxy DNS answer. Empty allow-net policy retains the existing allow-all behavior.
- 2026-07-04: M07 readiness is defined as a valid LocalSandbox `GuestReady` frame received over the established Windows virtio-serial control stream. `boot.status.json` records `state: "guest_ready"`, `success_definition: "localsandbox_guest_ready_frame_received_over_control_transport"`, elapsed readiness time, and the ready payload metadata. M09 allows the `file_range_io` readiness capability for chunked file transfers and still rejects unknown capabilities.

Guest ready handshake, exec/file API parity, muxing, mounts, networking, checkpoints, and Node packaging remain later milestones unless already supported by existing code paths.
